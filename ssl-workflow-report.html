📋 جزئیات سرویس‌ها

1️⃣ certbot_init (اجرا یک‌بار)

ویژگیمقدارتطابق با nginx
دامین ${DOMAIN_NAME} ✅ یکسان
Port 80 (standalone) ✅ nginx هم پورت 80 دارد
Challenge --preferred-challenges http ✅ HTTP-01
Volume letsencrypt → /etc/letsencrypt ✅ nginx از همین volume می‌خواند 
certonly --standalone --preferred-challenges http \
  -d ${DOMAIN_NAME} -m ${CERTBOT_EMAIL} \
  --agree-tos --non-interactive -vvv
مهم موفقیت: گواهی در /etc/letsencrypt/live/${DOMAIN_NAME}/ ذخیره می‌شود.
نکته سرویس یک‌بار اجرا می‌شود و پس از تکمیل متوقف می‌شود restart: "no".

2️⃣ certbot_renew (چرخه تمدید)

ویژگیمقدارتطابق با nginx
Webroot -w /var/www/certbot ✅ nginx همان را mount کرده
بازه sleep ${RENEW_INTERVAL:-12h} ✅ کافی برای جلوگیری از ریت لیمیت
ACME path --webroot -w /var/www/certbot ✅ nginx: location /.well-known/acme-challenge/
Volume certbot-webroot /var/www/certbot ✅ هر دو کانتینر mount کرده‌اند 
# Loop: هر ۱۲ ساعت اجرا می‌شود
while :; do
  certbot renew --webroot -w /var/www/certbot
  sleep 12h
done
تطابق کامل production.conf خط ۱۱-۱۳ دارای location /.well-known/acme-challenge/ با root /var/www/certbot است.
مهم این location قبل از redirect به HTTPS قرار دارد، در غیر این صورت ACME challenge با 301 مواجه می‌شود و fail می‌کند.

3️⃣ nginx (SSL Termination)

ویژگیمقدار
Imagenginx:1.27-alpine
Config/etc/nginx/templates/default.conf.template (production.conf)
SSL Cert path/etc/letsencrypt/live/${DOMAIN_NAME}/fullchain.pem
SSL Key path/etc/letsencrypt/live/${DOMAIN_NAME}/privkey.pem
DH Param/etc/letsencrypt/ssl-dhparams.pem
# بررسی تطابق مسیرهای گواهی در production.conf:
ssl_certificate         /etc/letsencrypt/live/${DOMAIN_NAME}/fullchain.pem;
ssl_certificate_key     /etc/letsencrypt/live/${DOMAIN_NAME}/privkey.pem;
ssl_trusted_certificate /etc/letsencrypt/live/${DOMAIN_NAME}/chain.pem;
ssl_dhparam             /etc/letsencrypt/ssl-dhparams.pem;
# بررسی volume mount در docker-compose.prod.yml:
volumes:
  - ./docker/nginx/nginx.conf:/etc/nginx/nginx.conf:ro
  - ./docker/nginx/conf.d/production.conf:/etc/nginx/templates/default.conf.template:ro
  - letsencrypt:/etc/letsencrypt:ro          # ← مسیر گواهی‌ها ✅
  - certbot-webroot:/var/www/certbot:ro       # ← ACME challenge ✅
  - ./nginx_configs/ssl-dhparams.pem:/etc/letsencrypt/ssl-dhparams.pem:ro  # ← DH ✅
✅ تطابق کامل مسیرهای گواهی در nginx دقیقاً منطبق بر محل ذخیره‌سازی certbot است.
⚡ HSTS فعال max-age=63072000; includeSubDomains; preload
🔒 TLS 1.2/1.3 با Mozilla Intermediate cipher suite

4️⃣ nginx_reloader (بارگذاری مجدد خودکار)

ویژگیمقدارتوضیح
Imagealpine:3.20 سبک و کافی برای inotify-tools
Dependencynginx (service_started) صبر نمی‌کند certbot_init کامل شود
Volume nginx certsletsencrypt:/etc/letsencrypt:ro ✅ برای دیدن تغییرات
Docker socket/var/run/docker.sock برای اجرای docker kill -s HUP 
# اسکریپت اصلی nginx_reloader:
apk add --no-cache inotify-tools docker-cli >/dev/null;
while :; do
  inotifywait -e create,modify,delete,move \
    -r /etc/letsencrypt/live >/dev/null 2>&1;
  echo "certs changed: reloading nginx";
  docker kill -s HUP shop-nginx-prod >/dev/null 2>&1 || true;
done
⚠️ ریسک امنیتی mount داکر سوکت (/var/run/docker.sock) دسترسی به هاست داکر می‌دهد.
✅ راه‌حل جایگزین در نسخه‌های جدیدتر nginx می‌توان از nginx -s reload داخل کانتینر nginx استفاده کرد.

📌 نکات تولید (Production Checklist)

✅ مزایای این معماری

  • 🔹 جدا بودن مسئولیت‌ها — هر کانتینر یک کار انجام می‌دهد
  • 🔹 Renew خودکار — نیازی به cron job یا اسکریپت خارجی نیست
  • 🔹 Reload خودکار — nginx_reloader با inotify تغییرات را تشخیص می‌دهد
  • 🔹 HSTS preload — امنیت حداکثری برای مرورگرها
  • 🔹 OCSP Stapling — سرعت بالاتر برای کلاینت‌ها

⚠️ نکات قابل بهبود

ریسک Docker socket mount در nginx_reloader — دسترسی به هاست داکر دارد راه‌حل: استفاده از nginx -s reload داخل کانتینر nginx (با docker exec)
توصیه متغیرهای env — حتماً DOMAIN_NAME و CERTBOT_EMAIL را در فایل .env یا environment تنظیم کنید در غیر این صورت متغیر خالی رفته و certbot fail می‌شود
توصیه Staging first — ابتدا با --staging تست کنید تا به ریت لیمیت Let's Encrypt نخورید بعد از اطمینان، --staging را بردارید
ریسک certbot_init روی پورت 80 — اگر nginx قبلاً روی 80 باشد، تداخل دارد خوشبختانه nginx به certbot_init وابسته است (depends_on: condition: service_completed_successfully)
اطلاعات frontend env — NEXT_PUBLIC_API_URL از طریق ${DOMAIN_NAME} تنظیم می‌شود مقادیر INTERNAL_API_URL=http://backend:8080 و NEXT_PUBLIC_API_URL=https://${DOMAIN_NAME}/api

⚙️ دستورات مفید

اجرای کل پروژه: 
docker compose -f docker-compose.prod.yml up -d
تست renewal به صورت دستی: 
docker exec shop-certbot-renew certbot renew --dry-run
مشاهده expiry گواهی: 
docker exec shop-certbot-renew openssl x509 -in /etc/letsencrypt/live/${DOMAIN_NAME}/fullchain.pem -noout -dates
Reload دستی nginx: 
docker kill -s HUP shop-nginx-prod
مشاهده لاگ‌های certbot: 
docker logs shop-certbot-init
docker logs shop-certbot-renew -f
مشاهده لاگ‌های nginx: 
docker compose -f docker-compose.prod.yml logs nginx -f
